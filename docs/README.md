# OctoAcme Project Management Docs

Welcome to the OctoAcme Project Management process documentation. This README is the landing overview for all OctoAcme process docs. More detailed documentation is available in the files listed below.

## Overview

OctoAcme runs projects as iterative, outcome-focused efforts with a clear lifecycle: Initiation → Planning → Execution → Release → Close & Retrospective. Work begins with a Project One-pager that captures the problem, goals, and success metrics, plus a lightweight decision gate to confirm stakeholder alignment and team availability before planning. Core artifacts include the one-pager/charter, prioritized backlog, release plan, acceptance criteria/Definition of Done, a risk register, and retrospective action items.

Workflows are practical and structured: planning includes a kickoff, prioritized backlog with acceptance criteria and estimates, and a documented Definition of Done. Execution is tracked via a project board (Backlog → Ready → In Progress → In Review → QA → Done), a small-PR-first pull request workflow with CI gating, and mandatory approvals. Team cadence includes daily standups for blockers and progress, weekly delivery syncs for updates and risk discussion, and sprint/milestone demos to show completed work.

Roles and communication are explicit: Project Manager coordinates delivery and communications, Product Manager defines outcomes and prioritizes work, Developers implement and test, QA validates acceptance, and Stakeholders provide inputs and approvals. Risks are managed in a simple Risk Register and escalations follow the path Team → PM → Product Lead → Sponsor. Quality practices include unit, integration, and end-to-end smoke tests, security scanning in CI, and a deployment checklist plus rollback playbook for releases.
